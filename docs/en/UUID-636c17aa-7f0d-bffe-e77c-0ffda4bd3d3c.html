Known flows launching process

With version 2.0.0 of the SDK, it’s possible to launch one of the standard CashComplete™ On-Device flows/services from within an SDK plugin flow. You can use the function to build an aggregated flow, or to add a custom action before or after a standard CashComplete™ On-Device flow.

You can use an aggregated flow, that is, a flow that is a sequence of other standard flows, for a number of different purposes:

  • Create a completely new function, for example, to create an Exchange as a combination of a standard deposit flow and a standard dispense flow. For more information, see the Exchange example.

  • Force users to follow a certain custom process. For example, someone wants to enforce a replenishment process where CIT always prints machine total, moves notes, empties boxes, and prints machine total again.

You can add a custom action when you want a new action before or after a standard flow, for example, to configure custom data before a deposit. In this case, you implement your own flow with the same icon and name as the standard flow and hide the standard flow by using blockPermssion(). From your flow, you launch the standard flow and add your custom actions before or after. For more information, see the CustomInfoDeposit example.

The main way of launching a standard flow is by calling executeKnownFlow() in IOJSystemManager, supplying an enum to indicate what known flow to launch. This method is going to block your plugin flow until the standard flow exits. Using this method, you can launch most flows that can be launched from select service. The enum contains more than 50 different known flows. For more information, see the API reference documentation for OJKnownFlow in IOJSystemManager.

Note

  • You can’t pass any data to the flows or receive any results.

  • To know what happened in the flow, you need to go to the IOJDatabaseManager to investigate new transactions stored.

A standard deposit flow can be launched with more control. You can launch a deposit with the method described above, but there is an optional way: IOJDepositManager has a method called runDepositFlow() that also blocks while a standard deposit is done. To use the runDepostiFlow(), you need to configure two parameters:

  • The transaction type to use

  • Whether or not to print a receipt

This method returns a money bag with the content that was deposited.
